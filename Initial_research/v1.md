# Invoice Processing Automation Architecture

## Summary

Build a Streamlit demo for PS-1 invoice processing that accepts a real invoice image/PDF, runs Azure Document Intelligence, uses an LLM only to produce a reusable mapping configuration from Azure’s raw tables/fields into a wide canonical invoice schema, then applies deterministic Python rules to approve, reject, or route to review.

Core story:

```text
Azure reads the document.
LLM maps Azure output to our schema and identifies invoice type.
Code builds normalized internal tables from Azure JSON using that mapping.
Code validates financial controls and makes the decision.
UI shows every stage and the final reasoning.
```

Key defaults:

- Azure Document Intelligence handles OCR/table extraction.
- LLM does not output final invoice facts directly.
- LLM outputs mapping config only.
- Python code applies the mapping config to Azure JSON.
- Approval logic is deterministic.
- Tolerance is configurable, default `0`, and variances do not auto-approve by default.
- Currency mismatch is flagged for review; no FX conversion.
- No full audit replay system.
- No metrics dashboard.
- No full reviewer override workflow for v1.

Target markdown file when implementation is allowed: `ARCHITECTURE.md`.

## Architecture

### Main Components

1. **Streamlit UI**
   - Upload invoice PDF/image.
   - Show live run stages.
   - Show original document preview.
   - Show Azure extracted fields/tables.
   - Show LLM mapping config.
   - Show normalized internal invoice record built by code.
   - Show deterministic rule checks.
   - Show final decision and explanation.
   - Show basic run history.

2. **Azure Document Intelligence Adapter**
   - Sends uploaded invoice to Azure prebuilt invoice model.
   - Stores raw Azure JSON.
   - Extracts:
     - document content
     - detected fields
     - detected tables
     - cell coordinates
     - confidence scores
     - page references
   - Does not make business decisions.

3. **Canonical Schema Dictionary**
   - Wide list of expected AP fields.
   - Covers common goods invoices, service invoices, progress invoices, credit notes, and project/contract invoices.
   - Used as the target vocabulary for the LLM mapping layer.

4. **LLM Mapping Layer**
   - Input:
     - Azure fields
     - Azure tables/cells
     - canonical schema dictionary
     - mapping instructions
   - Output:
     - invoice type
     - header field mappings
     - table mappings
     - row inclusion/exclusion rules
     - special record-shape rules if a table spans multiple rows
     - mapping confidence and warnings
   - Must not output final normalized invoice rows as facts.
   - Must not approve/reject invoices.

5. **Mapping Executor**
   - Pure Python.
   - Applies the LLM mapping config to the original Azure JSON.
   - Builds normalized internal tables.
   - Parses numbers, dates, currencies, percentages, and labels.
   - Preserves source references for each mapped value.

6. **Invoice Type Router**
   - Uses LLM-provided invoice type to select validation policy.
   - Supported v1 types:
     - `goods_po_invoice`
     - `service_po_invoice`
     - `professional_services_progress_invoice`
     - `credit_note`
     - `unknown_invoice`
   - If invoice type confidence is low, route to review.

7. **Rules Engine**
   - Deterministic Python checks.
   - No LLM decision-making.
   - Produces:
     - pass/fail/warn status
     - computed values
     - expected values
     - variance
     - reason
     - final decision contribution

8. **Data Store**
   - SQLite for local demo.
   - Stores:
     - vendors
     - purchase orders / contracts
     - PO lines or contract phases
     - receipt records where relevant
     - prior invoices
     - run history
     - raw Azure JSON
     - LLM mapping config
     - normalized invoice record
     - rule results
     - final decision

## LLM Mapping Contract

### Input

The LLM receives:

```text
1. Azure prebuilt invoice fields.
2. Azure tables with row/column indices and cell text.
3. Full OCR content if needed.
4. Canonical schema dictionary.
5. Instruction to output mapping config only.
```

### Output Shape

The LLM returns JSON like:

```json
{
  "invoice_type": "professional_services_progress_invoice",
  "invoice_type_confidence": "high",
  "header_mappings": {
    "invoice_number": {
      "source": "table_0",
      "row": 1,
      "column_under_header": "INVOICE NUMBER"
    },
    "project_reference": {
      "source": "table_0",
      "row": 1,
      "column_under_header": "OUR PROJECT NO."
    }
  },
  "table_mappings": [
    {
      "target_table": "progress_billing",
      "source": "table_1",
      "data_rows": "rows 2-4",
      "exclude_rows": ["totals row"],
      "columns": {
        "phase": 0,
        "contract_percent": 1,
        "contract_amount": 2,
        "previous_percent": 3,
        "previous_amount": 4,
        "current_percent": 5,
        "current_amount": 6,
        "remaining_amount": 7
      }
    }
  ],
  "mapping_warnings": []
}
```

### Hard Boundary

Allowed:

- map source columns to canonical fields
- identify invoice type
- describe table shape
- identify subtotal/total rows
- mark low-confidence mappings
- explain why a column maps to a field

Not allowed:

- approve invoice
- reject invoice
- invent missing values
- output final invoice line records as facts
- bypass source Azure cells

## Example: `hard_table_1.png`

Azure extracts:

- `VendorName = Fountainhead A+E`
- `InvoiceId = GALT-009`
- `InvoiceDate = 2013-08-31`
- `DueDate = 2013-09-30`
- `InvoiceTotal = 11812.50`
- `AmountDue = 11812.50`
- table 0 has `OUR PROJECT NO. = 2012-0001`
- table 1 has fee/progress summary
- table 2 has basic service time entries
- table 3 has additional service time entry with awkward multi-row shape
- table 4 has invoice total, previous balance, payments received, account balance

LLM mapping result:

```text
invoice_type = professional_services_progress_invoice

table_0:
INVOICE NUMBER -> invoice_number
INVOICE DATE -> invoice_date
PAYMENT DUE DATE -> due_date
OUR PROJECT NO. -> project_reference
BALANCE DUE -> amount_due

table_1:
phase -> progress_billing.phase
stipulated amount -> progress_billing.contract_amount
previous amount billed -> progress_billing.previous_amount
current value completed -> progress_billing.current_amount
remaining amount -> progress_billing.remaining_amount

table_2:
date -> service_lines.date
employee -> service_lines.person
code -> service_lines.service_code
description -> service_lines.description
hrs -> service_lines.hours
extension -> service_lines.line_amount

table_3:
multi-row additional service record
row with date/name gives phase/date/person
following row gives description/hours/line amount

table_4:
Invoice Total -> invoice_total
Previous Balance -> previous_balance
Payments Received -> payments_received
Account Balance -> account_balance
```

Then code builds normalized tables and validates:

```text
Progress current amount: $9,000.00
Basic Services subtotal: $1,250.00
Additional Services subtotal: $1,562.50
Computed invoice total: $11,812.50
Extracted invoice total: $11,812.50
Result: pass
```

Decision depends on internal data:

- If vendor approved, project `2012-0001` exists, no duplicate, currency matches, contract balance is sufficient, and totals reconcile: `Approved`.
- If project `2012-0001` is not found: `Needs Review`.
- If invoice `GALT-009` already exists for Fountainhead A+E: `Rejected`.
- If currency mismatches project currency: `Needs Review`.

## Validation Rules

### Universal Rules

Always run:

```text
required_fields_check
vendor_status_check
duplicate_invoice_check
currency_match_check
header_total_check
mapped_confidence_check
source_traceability_check
```

Decision defaults:

- Missing critical field: `Needs Review`
- Blocked vendor: `Rejected`
- Exact duplicate: `Rejected`
- Currency mismatch: `Needs Review`
- Low-confidence mapping on critical field: `Needs Review`

### Goods PO Invoice Rules

Run when `invoice_type = goods_po_invoice`:

```text
po_exists_check
po_status_check
vendor_po_match_check
line_quantity_check
unit_price_check
receipt_quantity_check
remaining_po_balance_check
```

Decision defaults:

- PO missing: `Needs Review`
- PO cancelled/closed: `Rejected`
- invoiced quantity exceeds received quantity: `Needs Review`
- invoice exceeds remaining PO balance: `Needs Review` or `Rejected` for severe mismatch

### Service PO Invoice Rules

Run when `invoice_type = service_po_invoice`:

```text
po_exists_check
vendor_po_match_check
service_code_check
service_period_check
rate_card_check_if_available
remaining_po_balance_check
```

Decision defaults:

- Missing service period: `Needs Review`
- Unknown service code: `Needs Review`
- Rate mismatch: `Needs Review`
- No rate card available: do not fail if PO amount still reconciles

### Professional Services Progress Invoice Rules

Run when `invoice_type = professional_services_progress_invoice`:

```text
project_or_contract_exists_check
vendor_project_match_check
progress_math_check
service_subtotal_check
invoice_total_reconciliation_check
remaining_contract_balance_check
duplicate_invoice_check
```

Specific checks:

```text
previously_billed + current_invoice + remaining = contract_amount
sum(current progress billing + service subtotals) = invoice_total
current_invoice <= remaining_contract_balance
```

### Credit Note Rules

Run when `invoice_type = credit_note`:

```text
negative_amount_check
original_invoice_reference_check
vendor_match_check
duplicate_credit_note_check
```

Decision defaults:

- Linked original invoice found and amounts reconcile: `Approved` or `Needs Review`
- Missing original invoice reference: `Needs Review`
- Positive total on credit note: `Needs Review`

### Unknown Invoice Rules

Run when `invoice_type = unknown_invoice` or confidence is low:

```text
required_fields_check
duplicate_check
vendor_check
basic_total_check
```

Decision:

```text
Needs Review
```

## Tolerance Policy

Tolerance is configurable but conservative.

Defaults:

```text
tolerance_amount = 0
tolerance_percent = 0
allow_tolerance_auto_approval = false
```

Behavior:

- Exact match + all controls pass: `Approved`
- Mismatch within configured tolerance: `Needs Review - Within Configured Tolerance`
- Mismatch outside tolerance: `Needs Review` or `Rejected`
- Duplicate, blocked vendor, cancelled PO: `Rejected`

Reasoning to present:

```text
Some companies require exact matching; some allow small rounding or tax variance. The system supports tolerance policies, but defaults to review rather than silent approval for money mismatches.
```

## Decisions

Final decision values:

```text
Approved
Needs Review
Rejected
```

Recommended decision hierarchy:

```text
Rejected beats Needs Review.
Needs Review beats Approved.
Approved only when all required checks pass.
```

Examples:

- exact duplicate: `Rejected`
- blocked vendor: `Rejected`
- missing PO/project reference: `Needs Review`
- low-confidence critical mapping: `Needs Review`
- amount variance inside tolerance: `Needs Review`
- all checks pass exactly: `Approved`

## Failure Handling

### Azure Failure

If Azure API fails:

```text
status = Needs Review
reason = Document extraction failed
fallback = allow cached Azure JSON for demo samples
```

### LLM Mapping Failure

If LLM fails or returns invalid JSON:

```text
status = Needs Review
reason = Mapping failed
show Azure extraction only
```

### Mapping Executor Failure

If mapping references missing tables/cells:

```text
status = Needs Review
reason = Mapping could not be applied to source document
show failed mapping path
```

### Rule Engine Failure

If a validation check errors:

```text
status = Needs Review
reason = Validation check failed unexpectedly
```

Do not crash the Streamlit app during demo.

## Acceptance Criteria

The build is complete when:

- A user can upload/select an invoice and run the process live.
- Azure extraction output is visible.
- LLM mapping config is visible.
- Internal normalized invoice record is built programmatically from Azure JSON.
- Rule checks are visible and deterministic.
- Final decision is one of `Approved`, `Needs Review`, `Rejected`.
- At least five demo scenarios run reliably.
- `hard_table_1.png` demonstrates messy table mapping without the LLM fabricating final rows.
- Tolerance is configurable but defaults to exact matching and review on variance.
- Currency mismatch is flagged for review.
- Run history persists across app refreshes.
